## Phantom Types for Compile-Time State Machines

A phantom type is a template parameter that appears only in the type tag — never in the data members. The parameter carries compile-time state information at zero runtime cost:

```cpp
struct Open   {};
struct Closed {};

template<typename State>
class FileHandle {
    int fd_;
    explicit FileHandle(int fd) : fd_(fd) {}

    // Only FileHandle<Open> has a read method
    friend FileHandle<Open>  open_file(const char* path);
    friend FileHandle<Closed> close_file(FileHandle<Open> h);
public:
    template<typename S = State>
    std::enable_if_t<std::is_same_v<S, Open>, ssize_t>
    read(void* buf, size_t n) { return ::read(fd_, buf, n); }
};
```

`FileHandle<Open>` and `FileHandle<Closed>` are distinct types — passing a `Closed` handle to `read()` is a compile error. State transitions (open → closed) return a different phantom instantiation. The runtime representation is identical — just an `int fd_`. The state exists only in the type.

This technique is also called "session types" in type theory. It guarantees protocol correctness (a function that requires a handshake cannot be called before `connect()`) at compile time, with zero runtime overhead.

## std::variant Internals: Aligned Storage and Discriminant

`std::variant<T1, T2, T3>` is implemented as:

```
[discriminant: uint8_t or similar]
[aligned storage: large enough for the largest alternative, aligned for the strictest]
```

The storage is `alignas(max(alignof(T1), alignof(T2), alignof(T3))) char[max(sizeof(T1), sizeof(T2), sizeof(T3))]`. Only one alternative occupies the storage at a time. The discriminant (index) indicates which one.

Accessing the wrong alternative is undefined behavior in the C++ standard. In debug builds with `_GLIBCXX_ASSERTIONS`, it throws `std::bad_variant_access`. In release builds without assertions, it accesses garbage memory silently.

`std::visit` dispatches via a jump table: a static array of function pointers, one per alternative, indexed by the discriminant. The dispatch is O(1) — one array lookup, one indirect call. No virtual dispatch, no if/else chain.

```
visit(visitor, variant_with_index_2):
  jump_table[2](visitor, &storage)  // direct call to lambda for type T2
```

The jump table is generated by the compiler at compile time. Each entry is a lambda instantiation specialized for one alternative type. If the variant has N alternatives, the jump table has N entries.

## The overloaded<Fs...> Visitor Pattern

The standard idiom for inline visitors:

```cpp
template<class... Ts>
struct overloaded : Ts... { using Ts::operator()...; };

// Deduction guide (makes the template argument deduction work from the lambdas)
template<class... Ts>
overloaded(Ts...) -> overloaded<Ts...>;
```

Usage:
```cpp
std::variant<int, double, std::string> v = 42;
std::visit(overloaded{
    [](int i)               { printf("int: %d\n", i); },
    [](double d)            { printf("double: %.2f\n", d); },
    [](const std::string& s){ printf("string: %s\n", s.c_str()); }
}, v);
```

`overloaded` inherits all three lambdas' `operator()` overloads. `std::visit` selects the correct one via overload resolution based on the active alternative's type.

The deduction guide is required because `overloaded` is a variadic class template — without it, you would need `overloaded<L1, L2, L3>{...}` with explicit types. The guide allows `overloaded{lambda1, lambda2, lambda3}` with deduction.

## Monadic Operations: and_then, or_else, transform

Monadic chaining eliminates the manual error-propagation boilerplate:

```cpp
// Without monads:
auto r1 = parse_int(s);
if (!r1) return unexpected(r1.error());
auto r2 = validate_positive(*r1);
if (!r2) return unexpected(r2.error());
auto r3 = compute_sqrt(*r2);
return r3;

// With monads:
return parse_int(s)
    .and_then(validate_positive)
    .and_then(compute_sqrt);
```

**`and_then(f)`:** If the current value is a success, applies `f(value)` and returns its result (which must itself be `Expected<U, E>`). If it is an error, propagates the error without calling `f`. This is the "bind" operation (Haskell's `>>=`).

**`transform(f)`:** If success, returns `Expected{f(value)}`. If error, propagates. Unlike `and_then`, `f` returns `U` not `Expected<U, E>`.

**`or_else(f)`:** If error, calls `f(error)`. If success, passes through. Used for fallback handling.

C++23 adds these to `std::optional` and `std::expected`. On GCC 11 (C++23 not available), implement them manually — see `examples/03_expected_chain.cpp`.

## User-Defined Literals

Literal operators allow you to attach strong types to numeric and string literals:

```cpp
struct Meters  { double value; };
struct Seconds { double value; };

Meters  operator""_m (long double v) { return {static_cast<double>(v)}; }
Seconds operator""_s (long double v) { return {static_cast<double>(v)}; }

auto dist = 1.5_m;   // Meters{1.5}
auto time = 3.0_s;   // Seconds{3.0}
```

Allowed parameter types for literal operators:
- `unsigned long long` — for integer suffixes
- `long double` — for floating-point suffixes  
- `const char*` (with or without `size_t`) — for string suffixes
- `char`, `wchar_t`, `char8_t`, `char16_t`, `char32_t` — for character suffixes

Suffixes must start with `_` for user-defined literals. Suffixes without `_` are reserved for the standard library (`1s` for `std::chrono::seconds`, `"hello"s` for `std::string`).

## The Spaceship Operator and Generated Comparisons

C++20's `<=>` operator (three-way comparison) generates all six comparison operators:

```cpp
struct Point {
    float x, y;
    auto operator<=>(const Point&) const = default;  // generates ==, !=, <, >, <=, >=
};
```

The return type communicates semantics:
- **`std::strong_ordering`:** `equal` means the values are substitutable in all contexts. For integer types.
- **`std::weak_ordering`:** `equivalent` does not guarantee substitutability. For case-insensitive strings.
- **`std::partial_ordering`:** Some values may be `unordered`. For IEEE floats (NaN comparisons).

When you default `<=>`, the compiler also generates `operator==` separately (C++20 changed this to allow symmetric `==` rewrites). If you define a custom `<=>`, you still need to define `==` explicitly or also default it.

Defaulted spaceship performs lexicographic member comparison. For the `Point` above, it compares `x` first, then `y`.

## explicit Constructors and Conversion Operators

Implicit conversions are a common source of bugs:

```cpp
struct Distance {
    Distance(double d) : val(d) {}  // implicit: Distance d = 5.0; works
    double val;
};

void set_radius(Distance d);
set_radius(5.0);   // compiles silently — is this km? m? mm?
```

With `explicit`:
```cpp
struct Distance {
    explicit Distance(double d) : val(d) {}
    double val;
};

set_radius(5.0);            // compile error
set_radius(Distance{5.0}); // explicit, intent is clear
```

The same applies to conversion operators. `operator bool()` without `explicit` allows `Distance d; if (d + 1)` to compile — the `Distance` converts to `bool`, which converts to `int`. With `explicit operator bool()`, the conversion only happens in boolean contexts (`if`, `while`, `!`, `&&`, `||`).

## std::any Internals and SBO

`std::any` stores a type-erased value with a runtime type tag (`std::type_info`). For small objects (GCC: typically <= 8 bytes on 64-bit, implementation-defined), it uses a small buffer optimization (SBO) to avoid a heap allocation. Large objects heap-allocate.

The internal vtable-like mechanism stores function pointers for copy, move, and destroy operations. `std::any_cast<T>` checks the stored `type_info` against `typeid(T)` and throws `bad_any_cast` on mismatch.

`std::any` is the type for "I genuinely don't know what type I'll store at compile time." For most cases, prefer `std::variant` (types known at compile time, exhaustive dispatch) or templates. `std::any` loses compile-time type safety entirely — use only when the set of types is truly open and runtime polymorphism on unknown types is required.
